# DevScope 开源生态情报站 - 项目要求

## 1. 项目概述
**DevScope** 是一个面向技术投资人和独立开发者的开源生态智能分析平台，旨在通过 AI 技术实时掌握 GitHub 项目健康度、技术趋势和竞争格局，帮助用户做出数据驱动的决策。

## 2. 数据来源
- **GitHub REST API**：获取仓库信息、Issues、PR、贡献者等。
- **Hacker News API**：获取社区讨论和热点。
- **OSS Insight API**：获取开源项目趋势和分析指标。

## 3. 三层架构设计
| 层级 | 名称               | 职责                                                                 |
|------|--------------------|----------------------------------------------------------------------|
| 1    | LLM直调            | 单次分析、结构化摘要、Embedding检索（已实现）                       |
| 2    | Workflow           | 固定流程编排（日报生成、周报汇总），通过 Langtum 平台实现           |
| 3    | Agent SDK          | 自主深度研究，多工具协作，基于 Claude Agent SDK 构建                |

## 4. 核心功能模块

### 4.1 GitHub 项目分析
- **输入**：GitHub 仓库 URL（如 `vercel/next.js`）
- **输出**：结构化分析报告，包含：
  - 健康度评分（0-100）
  - 活跃等级（high/medium/low/dead）
  - 关键指标（star增长率、issue解决率、贡献者多样性）
  - 风险因素列表
  - 机会列表
  - 投资建议（invest/watch/avoid）
- **要求**：输出必须符合 Zod Schema，强制使用 Claude 工具调用。

### 4.2 语义搜索与 RAG
- **数据采集流水线**：
  1. 调用 GitHub API 获取仓库基础信息和 README。
  2. 将 README 分块（500 token，重叠 50 token）。
  3. 生成 Embedding（使用 OpenAI `text-embedding-3-small` 或本地 bge-m3）。
  4. 存入 PostgreSQL（pgvector 扩展）。
- **检索功能**：
  - 支持自然语言查询，返回最相关的文本块及相似度分数。
  - 结合检索结果，由 LLM 生成带引用的综合回答。
- **接口**：tRPC 路由 `semanticSearch`，前端通过 TanStack Query 调用。

### 4.3 固定工作流（Langtum 平台）
- **工作流 1：每日项目健康度报告**
  - 触发：每天 8:00（cron）
  - 步骤：
    1. 获取用户关注列表。
    2. 对每个仓库执行数据采集和分析。
    3. 汇总排序，生成报告。
    4. 发送报告（邮件或前端展示）。
- **工作流 2：新项目快速评估**
  - 触发：用户提交 GitHub URL
  - 步骤（6 步）：
    1. 信息采集（GitHub + Hacker News）
    2. 代码质量分析
    3. 社区活跃度分析
    4. 竞品分析
    5. 风险评估
    6. 生成最终报告
- **对接方式**：后端通过 OpenAPI 调用 Langtum 平台，支持同步和异步模式。

### 4.4 深度研究 Agent（Claude Agent SDK）
- **目标**：给定任意技术主题，自主完成完整的竞争格局分析报告。
- **方法论**：
  1. 情报收集（调用 `repo-fetch`、搜索 Hacker News、论文等）。
  2. 竞品分析（并行分析多个项目，生成对比矩阵）。
  3. 深度洞察（技术趋势、生态成熟度、风险）。
  4. 报告生成（写入 `reports` 目录，Markdown 格式）。
- **工具集**：
  - `repo-fetch`：采集 GitHub 数据。
  - `repo-analyze`：LLM 分析健康度。
  - `search-hn`：搜索 Hacker News。
  - `db-query`：查询缓存。
  - `write-report`：生成报告。
- **前端集成**：通过 SSE 实时展示 Agent 思考过程。

### 4.5 多 Agent 协作
- **架构**：
Orchestrator Agent（任务分解、调度、汇总）
├─ Fetch Agent（数据采集）
├─ HN Agent（社区讨论）
└─ Competitor Analysis Agent（竞品分析）
↓
Report Agent（报告生成）

text
- **要求**：支持顺序、并行、层级编排，并可在关键节点加入人工确认（Human-in-the-Loop）。

### 4.6 CLI 技能（供 Agent 调用）
- **技能 1：`repo-fetch`**
- 输入：`owner/repo`，可选参数 `--include-issues`、`--include-commits`
- 输出：JSON 格式的仓库数据
- **技能 2：`repo-analyze`**
- 从 stdin 读取 `repo-fetch` 的输出，调用 Claude 分析，输出结构化结果。
- **技能 3：`report-generate`**
- 接收分析结果，支持模板（daily/weekly/investment），输出 Markdown/HTML。
- **设计原则**：单一职责、可组合（支持管道 `|`）、幂等性、JSON 输出。

## 7. 关键接口定义（示例）

### 7.1 项目分析 API（tRPC）
```typescript
// 输入
{ repo: string }  // 例如 "vercel/next.js"

// 输出
{
health_score: number;          // 0-100
activity_level: 'high' | 'medium' | 'low' | 'dead';
key_metrics: {
  stars_growth: number;        // 百分比
  issue_resolution_rate: number; // 0-100
  contributor_diversity: number; // 0-1
};
risk_factors: string[];
opportunities: string[];
recommendation: 'invest' | 'watch' | 'avoid';
}
7.2 语义搜索 API（tRPC）
typescript
// 输入
{ query: string; limit?: number }

// 输出
{
  answer: string;                // LLM 生成的回答
  sources: {
    text: string;                // 相关文本块
    repo: string;                // 来源仓库
    similarity: number;           // 相似度分数
  }[];
}
8. 注意事项
所有 AI 调用必须遵循结构化输出（工具调用），避免自由文本解析。

数据采集需处理 GitHub API 速率限制（使用令牌、重试、缓存）。

Agent 执行需记录详细日志，便于调试和审计。

前端实时进度展示：对于工作流和 Agent，使用 SSE 或轮询展示当前步骤。

所有模块必须有单元测试，核心逻辑覆盖率不低于 80%。
